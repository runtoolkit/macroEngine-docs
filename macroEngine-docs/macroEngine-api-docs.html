runtoolkit · Advanced Macro Engine

macroEngine
API Reference

A high-performance scripting and macro execution framework for Minecraft Java Edition datapacks. Provides fiber scheduling, modular systems, per-player cooldowns, custom events, and multi-version overlay support out of the box.

Namespace: macro Storage: macro:engine · macro:input · macro:output Minecraft 1.21.11 pack_format 94 Java Edition only
🗄
Storage Convention
Three canonical namespaces. Never write directly to macro:engine — it is internal state.
macro:input
Write call arguments here before invoking a function. Fields are read as $(field) macro variables.
macro:output
Return values written here by the callee. Read immediately — the next call will overwrite fields.
macro:engine
Internal runtime state (schedules, fibers, cooldowns, flags, events, wand/interaction binds). Read-only for user code.
⚠ Nesting rule: call macro:core/lib/input_push before a nested module call, macro:core/lib/input_pop after. Otherwise the outer call's input data is overwritten.
⚙
Calling Pattern
All AME functions use the with storage macro syntax to read their inputs.
✔
Standard call: Write fields to macro:input, then call with with storage macro:input {}
Compound call: Some functions read from a sub-key e.g. with storage macro:input sound
Inline call: Short functions accept inline NBT: $function macro:systems/log/info {message:"hello"}
.mcfunction · standard pattern
# 1. Write inputs to macro:input
data modify storage macro:input key set value "my_timer"
data modify storage macro:input func set value "mypack:on_expire"
data modify storage macro:input interval set value 40

# 2. Call the function with storage — all $(fields) are injected as macro args
$function macro:core/lib/schedule with storage macro:input {}

# 3. Read output
data get storage macro:output result

Delay-based and repeating function scheduling. Backed by the internal engine queue. Supports named (key-based) schedules for cancellation and renewal.

macro:core/queue
Throughput-limited work queue. Different from the delay scheduler — use this to process heavy function lists without causing lag spikes. Rate is configurable via queue/set_rate.
FNmacro:core/queue/pushEnqueue a function for rate-limited execution▾
Input (macro:input queue)
FieldTypeDescription
fnstringFully-qualified function path e.g. "mypack:do_thing"required
.mcfunction
data modify storage macro:input queue.fn set value "mypack:process_player"
function macro:core/queue/push with storage macro:input queue
ℹqueue/push vs core/lib/wait: queue/push is a throughput limiter (processes N items/tick). core/lib/wait is a time-based delay (fires after N ticks).
FNmacro:core/queue/flushProcess all queued items immediately▾
⚠Bypasses the rate limit. Use only when you need immediate guaranteed execution (e.g. cleanup on server stop).

macro:core/lib/ — IO Stack
input_push / input_pop save and restore all known macro:input fields to an internal stack, enabling safe nested module calls.
FNmacro:core/lib/input_pushSave current macro:input to stack▾
✔Saves all known input fields (player, key, func, delay, x/y/z, tag, sound, etc.) to macro:engine _input_stack[-1]. Always pair with input_pop.
.mcfunction · nested call
# outer call is in progress — save its inputs
function macro:core/lib/input_push

# now safely write new inputs and call a nested function
data modify storage macro:input key set value "inner_flag"
function macro:systems/flag/get with storage macro:input {}

# restore outer inputs
function macro:core/lib/input_pop
FNmacro:core/lib/input_popRestore macro:input from stack▾
⚠Each input_push must have exactly one matching input_pop. Mismatched calls corrupt the input stack and produce silent bugs.
